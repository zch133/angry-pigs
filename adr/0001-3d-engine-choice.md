# ADR-0001: 使用 Three.js + cannon-es 作为 3D 引擎

## Status
Accepted

## Date
2026-07-07

## Context
游戏需要 3D 立体版本，在浏览器中运行，部署在 GitHub Pages。用户不懂代码，需要能独立完成所有部署和测试工作。需要物理引擎处理弹射、碰撞、重力。

## Decision
使用 Three.js 做 3D 渲染，cannon-es 做物理模拟。纯前端方案，无后端依赖。

## Consequences
- 正面：浏览器原生支持，无需安装插件；Three.js 社区大、文档全；cannon-es 专门做 3D 刚体物理；纯静态部署到 GitHub Pages 零成本。
- 负面：3D 性能受限于浏览器，复杂场景可能掉帧；需要手动处理 Three.js 和 cannon-es 的物体同步。

## Alternatives considered
- Unity WebGL：需要 Unity 编辑器构建导出，用户无法操作；包体大，加载慢。
- Babylon.js：与 Three.js 同级别，但社区稍小。
- PlayCanvas：需要注册平台账号，用户不想注册额外服务。

# ADR-0002: 3D 模型用代码精细建模，不使用外部模型文件

## Status
Accepted

## Date
2026-07-07

## Context
用户要求"更真实的 3D 模型"。考虑了 AI 生成模型（Meshy 等方案），但需要注册外部服务，用户拒绝。需要一种用户零参与、能完全由开发者控制的方式。

## Decision
使用 Three.js 的 LatheGeometry、ExtrudeGeometry、SphereGeometry 等原生几何体组合精细建模。配合平滑着色、渐变颜色、简单动画做出圆润卡通风格。

## Consequences
- 正面：零外部依赖，不需要模型文件；完全可控，能调试修改；风格统一。
- 负面：精细度有上限，做不到毛发纹理、骨骼动画、面部肌肉表情；比专业建模师做的简陋。
- 能做到：圆滑球体身体、锥形嘴/鼻子、球体眼睛、扁平耳朵/翅膀、颜色和材质区分、简单表情、尾巴卷曲、待机动画（晃动/翅膀微动）。

## Alternatives considered
- AI 生成 3D 模型（Meshy）：需要注册，用户拒绝。
- 免费 3D 模型库下载（Sketchfab 等）：风格不统一，版权风险。
- 纯几何体简单组合（方案8原版）：精细度不够，用户拒绝。

# ADR-0003: 鸟的碰撞规则——飞行时无碰撞但不穿过，掉落时全物理碰撞

## Status
Accepted

## Date
2026-07-07

## Context
用户要求鸟与建筑物"无碰撞但不穿过"，且鸟被击中后掉落"符合真实物理情况"。这需要在同一只鸟上区分两种碰撞行为。

## Decision
鸟有两个物理状态：
1. **活着状态**（飞鸟飘移 / 站鸟跑动）：鸟与建筑物之间不产生物理碰撞，但鸟的运动轨迹会被建筑物的包围盒限制——碰到建筑物边界就转向，效果上等同于"不穿过"。
2. **掉落状态**（被猪击中后）：鸟切换为完整物理刚体，受重力影响，与建筑物和地面产生碰撞，会弹开或停在建筑物上。

## Consequences
- 正面：符合用户要求的"不穿过但也不碰撞"和"掉落时真实物理"。
- 负面：需要在击中瞬间切换鸟的碰撞属性，有状态管理复杂度；活着的鸟的避障逻辑需要额外处理。

## Alternatives considered
- 飞行时直接穿过建筑物：用户明确拒绝。
- 飞行时物理碰撞但建筑物不损坏：用户明确拒绝（"鸟就不能撞到建筑物"）。

# ADR-0004: 云端存档延后，第一版只做本地存档

## Status
Accepted

## Date
2026-07-07

## Context
用户有多设备同步需求，想跟朋友分享游戏。云端存档需要后端服务器，GitHub Pages 不支持后端。

## Decision
第一版只做 localStorage 本地存档。云端存档作为后续升级，等核心玩法验证可玩性后再加。

## Consequences
- 正面：降低首批交付复杂度，先跑通核心玩法；不需要搭建后端基础设施。
- 负面：换设备/清缓存会丢进度；朋友各自的进度独立，无法同步。

## Alternatives considered
- 第一版就做云端存档（Vercel + Supabase）：增加基础设施复杂度，拖慢核心玩法交付。
